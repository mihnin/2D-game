## Architecture

**Boxing Liberation** is a 2D side-scrolling beat-em-up built on HTML5 Canvas with vanilla ES6 modules (no bundler). Jest + Babel transforms ESM for testing.

### Key Layers

- **`js/main.js`** — Entry point. Creates `Game` class that wires together `AssetLoader`, `InputManager`, `SceneManager`, and `GameLoop`. Assets are loaded from root directory (`avatar.png`, `solder.png`, `fone.png`) as whole sprite sheets.
- **`js/config/constants.js`** — All gameplay tuning values (HP, speeds, level thresholds, combo multipliers, colors). Change balance here.
- **`js/config/spriteData.js`** — Frame coordinates for sprite sheets. Hero uses uniform 4x4 grid (154x256 per frame). Enemy uses explicit `{x, y, w, h}` rectangles per frame. Backgrounds are 3 vertical regions of `fone.png`.

### Scene System

Scenes have `enter(data)`, `exit()`, `update(dt)`, `render(ctx)` lifecycle. SceneManager handles switching. Five scenes: `menu`, `game`, `pause`, `gameover`, `victory`. The pause scene holds a reference to the game scene to render behind its overlay.

### GameScene Orchestration

`GameScene.enter()` creates a fresh `Player`, resets all systems, and wires up an `EventBus`. The update loop: player input → player update → enemy spawn/update → collision detection → combat processing → camera follow → particles. The EventBus connects combat outcomes to progression (`enemyKilled` → `LevelSystem.checkProgression` → potential `levelUp` or `victory` emit → scene switch).

### Entity State Machines

`Player` and `Enemy` each manage their own state (idle/walking/jumping/punching/hurt/dead). State determines which animation plays and what actions are allowed. Player handles its own gravity in `update()` rather than using PhysicsSystem. Entities carry an `AnimationController` from `AnimationSystem.js` that tracks current frame.

### Sprite Rendering and Background Removal

Sprites are drawn directly from source sprite sheets via `ctx.drawImage(img, sx, sy, sw, sh, dx, dy, dw, dh)` — no pre-cut files at runtime. The `AnimationController.getCurrentFrameData()` returns either `{type:'grid', row, col}` (hero) or `{type:'rect', frame:{x,y,w,h}}` (enemy), and GameScene calls the appropriate drawImage variant. Pre-cut sprite PNGs exist in `assets/sprites/` for reference (generated by `node tools/cut-sprites.js`).

The source PNGs (`avatar.png`, `solder.png`) have white/checkerboard backgrounds **and text labels** (row headers like "ROW 1:", frame numbers like "1.1") instead of true alpha transparency. `AssetLoader.removeBackground(key)` uses **flood fill from image edges** + fringe cleanup at init time. The spriteData.js coordinates are carefully tuned to reference only sprite body regions, excluding text labels:

- `solder.png` walkRight: y=53-188 (text labels at y=20-44)
- `solder.png` walkLeft: y=535-669 (deathLeft_prone tail + text at y=460-534)
- Use `debug-sprites.html` in the browser to visually analyze sprite sheet coordinates

**Important:** When modifying enemy sprite coordinates, use pixel analysis with maxRun>=15 threshold to distinguish real sprite content from small text labels. Do NOT apply background removal to `fone.png` backgrounds.

### Combat → Score → Level Flow

`CollisionSystem` returns hit lists → `CombatSystem.processAttackHits()` applies damage, tracks combo, emits events → `LevelSystem.checkProgression()` checks score thresholds and can advance multiple levels in one call (uses `while` loop) → emits `levelUp` which reconfigures `SpawnSystem` difficulty and swaps background region.

### Test Setup

`tests/setup.js` mocks Canvas 2D context, `requestAnimationFrame`, `performance.now`, and `Image`. Tests import source modules directly; Babel (`babel.config.json`, test env only) transforms ESM to CJS for Jest. The `sharp` dependency is used only by `tools/cut-sprites.js` (Node.js sprite cutter), not at runtime.
